<!-- SQLLogicTest format patterns used in this project -->

SQLLogicTest type specifiers:
- `query T` → VARCHAR/text column
- `query I` → INTEGER column (also matches BIGINT)
- `query R` → REAL/DOUBLE column
- `query D` → DATE column (returns YYYY-MM-DD format strings in assertions)
- `query TT` → two text columns
- `query TI` → text + integer columns

Pattern for `create_semantic_view` (6-arg):
```sql
SELECT create_semantic_view(
    'view_name',
    [{'alias': 'o', 'table': 'table_name'}],  -- tables
    [],                                          -- relationships
    [{'name': 'dim', 'expr': 'col', 'source_table': 'o'}],  -- dimensions
    [],                                          -- time_dimensions
    [{'name': 'metric', 'expr': 'sum(col)', 'source_table': 'o'}]  -- metrics
);
```

Pattern for `explain_semantic_view`:
```sql
-- Returns multiple rows of VARCHAR, each row is one line of EXPLAIN output
-- The expanded SQL appears in the output (look for SELECT or WITH keyword)
FROM explain_semantic_view('view_name', dimensions := ['dim'], metrics := ['metric'])
```

Pattern for typed output test (BIGINT metric):
```sql
-- Use query I when the metric is expected to be BIGINT
query I
SELECT revenue FROM semantic_view('view_name', metrics := ['revenue'])
----
<integer_value>
```

Pattern for DUCKDB typed output DATE dimension:
```sql
query D
SELECT order_date FROM semantic_view('view_name', dimensions := ['order_date'])
----
2024-01-01
```

The `require semantic_views` directive at the top of each test file loads the extension.

Tip comment format (add to phase4_query.test header or a dedicated comment block):

```
# plan (TABLE_FUNCTION node). Use explain_semantic_view('view', ...) to see
# the expanded SQL that semantic_view executes internally.
```
