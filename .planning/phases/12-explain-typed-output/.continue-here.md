## Next: Add Phase 13 — Type-mapping and PBT for typed column dispatch

The HUGEINT bug revealed a gap: `type_from_duckdb_type_u32`, `parse_typed_from_str`, and `write_typed_column` must agree on slot sizes, but there are no unit tests enforcing this invariant.

**Command to run first in fresh session:**
```
/gsd:add-phase Type-mapping and property-based tests for typed column dispatch
```

**What Phase 13 should cover (thoroughly):**

1. **Type consistency table test** — for every `ffi::DUCKDB_TYPE_*` constant, assert that:
   - `type_from_duckdb_type_u32(t)` returns a LogicalTypeId whose slot size matches
   - `parse_typed_from_str(s, t)` returns the correct TypedValue variant
   - The variant and the declared type agree on byte width
   - Specifically catches: HUGEINT→BIGINT, DECIMAL→VARCHAR, UHUGEINT→BIGINT

2. **PBT with proptest on `parse_typed_from_str`** — generate arbitrary:
   - i64 values (positive, negative, zero, MIN, MAX) → assert BIGINT parse is stable and lossless
   - i32 values → assert INTEGER parse
   - f64 values (NaN, Inf, subnormals, very large) → assert DOUBLE parse / fallback
   - Date strings (valid YYYY-MM-DD, leap days, epoch, year 0, far future) → assert DATE parse
   - Empty string → always TypedValue::Null for numeric types
   - Non-numeric strings → assert fallback to Null (not panic)

3. **PBT on parse→write roundtrip** — generate values, write to a real DuckDB table via semantic_view, read back, assert equality. May need a test harness with an in-process DuckDB instance.

4. **Integration test with large sums** — `sum()` on 100,000 rows to exercise real HUGEINT values that would have shown garbage with the old code.

**Files to modify:** `src/query/table_function.rs` (add test module), `Cargo.toml` (add proptest dev-dependency)
</remaining_work>
