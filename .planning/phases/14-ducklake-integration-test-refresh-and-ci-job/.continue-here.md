## Pre-existing bug: `test/sql/phase4_query.test:300` fails

**Symptom**: `semantic_view('typed_date_test', dimensions := ['event_date'], metrics := ['total_count'])` returns `2024-02-01 | 0` instead of `2024-02-01 | 3` for February's sum.

**Root cause** (fully diagnosed):
- `try_infer_schema` (LIMIT 0) infers `sum(event_count)` type as **HUGEINT** (DuckDB plans standard `sum()` as HUGEINT at LIMIT 0 time)
- At runtime, DuckDB's optimizer uses **`sum_no_overflow`** → returns **BIGINT** (8 bytes/value)
- `column_type_ids` stores HUGEINT (21); the HUGEINT reader reads **16 bytes/value** (i128)
- Actual column is BIGINT (8 bytes/value): data = [5i64, 3i64] = 16 bytes total
  - Row 0: reads bytes 0–15 as i128 → lower 64 bits = 5 ✓ (accidentally correct)
  - Row 1: reads bytes 16–31 as i128 → **out of bounds** → 0 ✗
- Write side is also wrong: `write_typed_column` with HUGEINT type_id writes i128 to a BIGINT output vector (declared as BIGINT via `type_from_duckdb_type_u32(HUGEINT) → BIGINT`)

**Fix** (identified, not yet implemented):
Normalize HUGEINT → BIGINT (and UHUGEINT → UBIGINT) in `column_type_ids` at bind time. Two locations:

1. **`src/query/table_function.rs` bind() fallback path** (line ~432):
```rust
// CHANGE:
let type_ids: Vec<u32> = types.iter().map(|t| *t as u32).collect();
// TO:
let type_ids: Vec<u32> = types.iter().map(|t| normalize_hugeint(*t as u32)).collect();
```

2. **`src/ddl/define.rs` DDL-time inference** (line ~156):
```rust
// CHANGE:
parsed.def.column_types_inferred = types.iter().map(|t| *t as u32).collect();
// TO:
parsed.def.column_types_inferred = types.iter().map(|t| normalize_hugeint(*t as u32)).collect();
```

Where `normalize_hugeint` is:
```rust
fn normalize_hugeint(t: u32) -> u32 {
    const HUGEINT: u32 = ffi::DUCKDB_TYPE_DUCKDB_TYPE_HUGEINT;
    const UHUGEINT: u32 = ffi::DUCKDB_TYPE_DUCKDB_TYPE_UHUGEINT;
    const BIGINT: u32 = ffi::DUCKDB_TYPE_DUCKDB_TYPE_BIGINT;
    const UBIGINT: u32 = ffi::DUCKDB_TYPE_DUCKDB_TYPE_UBIGINT;
    match t {
        HUGEINT => BIGINT,
        UHUGEINT => UBIGINT,
        _ => t,
    }
}
```

This is consistent with `type_from_duckdb_type_u32` which already maps HUGEINT output → BIGINT.
