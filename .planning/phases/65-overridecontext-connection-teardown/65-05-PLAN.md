C++ scalar registration helper (NEW in cpp/src/shim.cpp, parallels Plan 02's sv_register_table_function):

```cpp
extern "C" bool sv_register_scalar_function(
    duckdb_database db_handle,
    const char *name,
    const sv_table_function_signature_t *sig,    // reused for arg types
    int return_type_id,                          // LogicalTypeId of return value
    sv_scalar_invoke_fn_t rust_invoke,           // (conn, input_chunk, output_chunk) -> bool
    sv_drop_rust_state_fn_t rust_drop_state);    // optional per-registration state (null for stateless)

// Trampoline (pseudo — exact ExpressionState::GetContext() access path validated by Task 0 spike):
static void sv_scalar_trampoline_invoke(DataChunk &input, ExpressionState &state, Vector &result) {
    auto &extra = state.expr.bind_info->Cast<SvRustScalarExtraInfo>();
    Connection probe(*state.GetContext().db);   // access path locked by SCALAR-PATH-RC0/RC1
    auto conn = reinterpret_cast<duckdb_connection>(&probe);
    char *error_msg = nullptr;
    bool ok = extra.rust_invoke(conn, &input, &result, &error_msg);
    if (!ok) {
        string msg = error_msg ? string(error_msg) : "scalar invoke failed";
        if (error_msg) sv_free_cstring(error_msg);
        throw ExecutorException(msg);
    }
}
```

semantic_view expansion routing (BLOCKER 3 LOCKED — option (a)):

The expanded SELECT that produces the aggregate result runs on the CALLER's connection via the table-function's native plan path. The current `execute_sql_raw` call against `query_conn` (H2 long-lived) is a v0.8.0 implementation detail that B-prime retires.

Rationale for locking option (a) over option (b):
1. The expanded SELECT then participates in the caller's session/transaction (consistent with B-prime's principle that all SQL runs on the caller's conn).
2. There is no extension-owned long-lived state to clean up — no `make_uniq<Connection>` on bind data, no second registration helper variant.
3. It's symmetric with the write path's "rewrite-to-native onto caller's binder" pattern (D-20) — the same Phase 58 mechanism that DDL writes use.
4. `execute_sql_raw` retirement from the `semantic_view` call path is the correct cleanup — it was the C-API-on-stashed-conn pattern that B-prime eliminates.

Mechanism: `semantic_view`'s bind body (running on a per-call C++ probe via sv_trampoline_bind) reads the view definition, builds the expansion, and returns a `ParserExtensionPlanResult` / `TableRef` (constructed from the rewritten SELECT) that the binder drives onto the caller's conn. This is the SAME mechanism Plan 03's sv_plan_function uses for DDL writes — only the SQL kind differs (SELECT vs INSERT/DELETE). The implementation reuses the existing Phase 58 helper that converts a SQL string into a TableRef for the parser-extension plan path (search `cpp/src/shim.cpp` + the surrounding Phase 58 code for the SQL→TableRef converter).

Consequence: `semantic_view` and `explain_semantic_view` do NOT need any C++ helper variant beyond `sv_register_table_function` (Plan 02). They register through the SAME helper Plan 04's read functions use. The "separate trampoline with query-scoped Connection" / "trampoline flag" alternatives from earlier drafts are DROPPED — option (a) makes them unnecessary because there is no execute-time conn requirement at all.

QueryState retirement:
