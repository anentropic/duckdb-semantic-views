**Cascaded Plan 02a's new C ABI into `init_extension`: 17 `sv_register_*` extern declarations + 17 call sites in `src/lib.rs` now allocate per-site error buffers, propagate the buffer text into returned `Err(...)` messages via a centralised `decode_register_err_buf` helper, and the Wave-0 stub `tests/registration_error_surfaces.rs` is populated with the D-05 + D-08/D-09 structural assertions.**

## Accomplishments

- **`src/lib.rs::init_extension` cascade landed** — 17 `sv_register_*` extern declarations gained the trailing `(error_buf: *mut c_char, error_buf_len: usize)` pair; 17 call sites allocate a fresh `[0u8; 1024]` buffer, pass it to the C helper, and on failure decode the diagnostic via `decode_register_err_buf` and embed it in the returned `Err(...)`. `sv_register_parser_hooks` itself is deliberately left untouched (Plan 10 / WR-06 owns that signature change).

## Decisions Made

- **Structural FFI-invariant test (over direct-FFI invocation).** See "key-decisions" above. The plan-literal direct-FFI test cannot be realised in the current crate layout without invasive build-system surgery; the structural alternative is the accepted fallback per the plan's own guidance and mirrors `tests/no_long_lived_conn.rs`.
- **`sv_register_parser_hooks` extern signature preserved.** Plan 10 (WR-06) explicitly owns the parser-hooks signature update together with the runtime layout probe. Touching it here would either duplicate that work or block Plan 10 on a partial state. Comment added to the extern block records the deliberate omission.
- **Per-site `let mut error_buf` allocation.** Could have been hoisted once at the top of `init_extension` and zeroed between calls. Chose the per-site shape because (a) each call site is now self-contained for code review, (b) no risk of stale-text reads on a subsequent registration's failure path, (c) the 17 1024-byte stack allocations are trivial compared to the duckdb_open they're paired with.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Blocking] Structural test path instead of literal direct-FFI invocation**

- **Found during:** Task 2 (`cargo test --test registration_error_surfaces` after writing a direct-FFI version)
- **Issue:** The plan's literal acceptance criterion asks for a Rust integration test that invokes `sv_register_table_function` with `init_cb = None`. That symbol is only compiled into the binary when `--features extension` is active (build.rs only compiles `cpp/src/shim.cpp` under that gate). Under `--features extension` the `duckdb/loadable-extension` Cargo feature replaces all `libduckdb-sys` C API calls with no-op stubs, so `duckdb_open(":memory:")` no longer returns a usable `duckdb_database` handle. The test cannot get a database to pass to `sv_register_table_function`. Dual-compiling `shim.cpp` under the default `bundled` feature was investigated: it would need build.rs to be reworked to expose `cc` as a non-`extension`-gated build-dep, and the resulting binary would attempt to link two independently-compiled DuckDB amalgamations (libduckdb-sys's bundled copy + this crate's `cpp/include/duckdb.cpp`), almost certainly producing C++ symbol collisions. The scope of such a build-system rework is firmly outside Plan 02b's mandate.
- **Fix:** Took Plan 02b's explicit fallback path: "adapt by gating with `#[cfg(not(feature = "extension"))]` or by re-using a Rust-side helper exposed in `src/lib.rs` for testing — discretion to whichever resolves cleanly given the existing crate structure". Implemented a structural test that parses `cpp/src/shim.cpp` from the filesystem and asserts the D-05 refusal text + D-08/D-09 ABI shape. Behavioural coverage of the same path lands via `just test-sql`, which exercises `LOAD semantic_views` end-to-end — any registration failure now surfaces through Task 1's error_buf cascade.
- **Files modified:** tests/registration_error_surfaces.rs
- **Committed in:** `06403a3` (Task 2 commit)
