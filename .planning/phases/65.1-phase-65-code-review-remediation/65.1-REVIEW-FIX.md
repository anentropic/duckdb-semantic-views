## Fixed Issues

### CR-01: `sv_register_parser_hooks` leaks `OverrideContext` on idempotent re-LOAD

**Files modified:** `cpp/src/shim.cpp`, `tests/parser_hook_idempotent.rs`
**Commits:** 7a9fc24 (primary), 4055196 (structural-test anchor refinement)
**Applied fix:**

Moved the `void *rust_state = sv_make_override_context();` allocation
from above the dedup loop into the `if (!already_registered) { ... }`
block. The original Plan 12 patch had moved the
`ParserExtension::Register` call and the `SemanticViewsParserInfo`
allocation behind the guard but left `sv_make_override_context()`
unconditional — so every redundant `LOAD semantic_views` still leaked
one heap-allocated `Box<OverrideContext>`. The struct is empty so the
per-LOAD cost is small, but the growth is unbounded across a long-lived
host process — the precise leak shape WR-09 was meant to fix.

Updated `tests/parser_hook_idempotent.rs` Assertion 5 to pin BOTH
allocations (`new SemanticViewsParserInfo(` AND
`void *rust_state = sv_make_override_context(`) behind the guard. The
prior assertion only checked `SemanticViewsParserInfo`, which is why
the regression escaped structurally. The follow-up commit 4055196
anchored the test on the assignment-prefixed form
(`void *rust_state = sv_make_override_context(`) rather than the bare
function name, so the find() can't be confused by comment references
to the function name appearing before the guard.

### WR-01: `sv_register_table_function` dereferences `arg_types` without null guard

**Files modified:** `cpp/src/shim.cpp`
**Commit:** 2354706
**Applied fix:**

Added the symmetric `(arg_count > 0 && arg_types == nullptr)` null
guard to both `sv_register_table_function` (line 631) and
`sv_register_scalar_function` (line 721), alongside the existing
null guards for `db_handle / name / exec_cb / init_cb /
DatabaseWrapper`. Diagnostic message surfaces via the same
`write_err` snprintf channel and ABI-stable `error_buf`, matching the
D-05/D-08/D-09 hardening pattern Phase 65.1 Plan 02a already applied.

The sister helpers `sv_register_explain_semantic_view_impl` and
`sv_register_semantic_view_impl` construct their args vectors from
stack literals, so they're not affected (matches the finding's note).

### WR-02: `sv_semantic_view_function` silently truncates column-count mismatch

**Applied fix:**

Replaced `idx_t n_cols = std::min<idx_t>(chunk->ColumnCount(),

This restores the legacy Rust impl's fail-loud semantics (which had a
`Mutex + col_count` check) without re-introducing the long-lived-state
concern the Phase 65 migration was retiring.

### WR-03: ENUM-related stale comment in `sv_resolve_output_logical_types`

**Files modified:** `cpp/src/shim.cpp`
**Commit:** de05dd0 (folded into WR-07 commit)
**Applied fix:**

### WR-04: `extract_quoted_string` / `extract_view_comment` / `extract_single_quoted` corrupt multi-byte UTF-8

**Files modified:** `src/parse.rs`
**Commit:** 7a2ea28
**Applied fix:**

Rewrote all three helpers to walk the input as a `char` stream via
`char_indices()` rather than as `&[u8]` with the `bytes[pos] as char`
cast. The cast interprets each byte as a Unicode scalar value (Latin-1
supplement region U+0080..U+00FF for non-ASCII) which silently
mangled Cyrillic, CJK, emoji, em-dash, smart-quote, and accented Latin
payloads. Preserved the SQL `''` escape semantics via iterator clone
+ `matches!(peek.next(), Some((_, '\'')))` peek pattern. Byte-offset
arithmetic adjusted so the consumed-length return values remain
correct (the closing quote is always one ASCII byte, so the offset
arithmetic stays simple).

Added 11 unit tests covering Cyrillic, CJK, emoji, em-dash, smart
quotes, and accented Latin payloads across all three helpers, plus
the `''` escape interaction with non-ASCII surroundings. Tests pass:
`test result: ok. 11 passed; 0 failed`.

The finding's note about this being a pre-existing bug surfaced by
Phase 65.1 Plan 07's FileSystem-direct YAML read path is accurate
and the fix protects the FROM YAML FILE Unicode-path case that the
review specifically called out.

### WR-05: `parse_string_list` returns flat `None` with no FFI-shape detail

**Files modified:** `src/query/explain.rs`, `src/query/table_function.rs`
**Commit:** 711a44a
**Applied fix:**

Changed both per-file `parse_string_list` helpers from
`Option<Vec<String>>` to `Result<Vec<String>, String>` with detailed
diagnostics:
- `"null buffer but len=N (FFI shape drift)"`
- `"buffer too short for count prefix: len=N (expected >= 4)"`
- `"expected u32 at offset N of M (truncated)"`
- `"element I of N declares length L but only K bytes remain at offset O"`
- `"trailing N bytes after count C (consumed K of L)"`

Updated all six call sites in the two dispatchers to render the
detail inline: `"malformed `dimensions` payload: {detail}"`, etc.
This matches the diagnostic shape of the C++ `sv_parse_varchar_payload`
helpers, eliminating the diagnostic asymmetry the finding flagged.

### WR-06: `sv_parser_override` discards Rust error on Err path

**Files modified:** `src/parse.rs`
**Commit:** e040ddb
**Applied fix:**

Renamed `Err(_err)` to `Err(err)` and added
`write_error_to_buffer(error_out, error_out_len, &err.message)` on the
rc=2 (defer-to-default-parser) path. The buffer is heap-allocated and
the write cost is sub-microsecond; the existing `write_error_to_buffer`
helper respects the caller's capacity and truncates at a char
boundary (proptest-verified by
`write_error_to_buffer_truncates_at_char_boundary`), so no new safety
concern. Future tooling (debug-build flag, tracing hook) can now
surface both errors without another ABI change.

The C++ side currently ignores the rc=2 channel per the Phase 62
design; this fix keeps the channel populated for the case the finding
describes (non-deterministic `rewrite_to_native_sql` failure paths
producing different errors on the second sv_parse_stub invocation).

### WR-07: Three Connection allocations per `semantic_view(...)` invocation

**Files modified:** `cpp/src/shim.cpp`
**Commit:** de05dd0
**Applied fix:**

Changed `sv_resolve_output_logical_types`'s signature from
`(ClientContext &context, ...)` to `(Connection &probe, ...)` and
updated `sv_semantic_view_bind` to pass its already-open `probe`
Connection down. One Connection now covers both the Rust dispatcher's
catalog reads AND the bind-time LIMIT-0 DECIMAL/LIST probe — same
transaction/catalog snapshot for both, and one fewer ctor/dtor pair
through `ConnectionManager::AddConnection`.

**Not addressed (documented in commit body):** the third Connection in
`sv_semantic_view_init_global` (line 2463). That call site is invoked
AFTER bind completes, so the bind's Connection is already destroyed;
consolidating those two would require stashing a Connection in
BindData and reasoning about its lifetime across the bind →
init_global → exec boundary, which the DuckDB extension API doesn't
make easy. The WR-07 finding's broader "three Connection" framing is
reduced to two; the most impactful consolidation (the two Connections
that should agree on transaction/catalog state) is achieved.

The finding's deeper concern about visibility of uncommitted writes
remains documented at the call site — `Connection(*context.db)`
doesn't satisfy a "see caller's uncommitted writes" contract regardless
of whether we open one or two, so that's a separate follow-up.
